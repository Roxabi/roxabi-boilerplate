### Summary
The P2 partition (Users, Organizations, Tenant) shows clear separation at the controller/service level but suffers from leaky repository boundaries. The `UserRepository` interface and its Drizzle implementation manage tables from four sibling domains (`organizations`, `members`, `invitations`, `sessions`, `roles`), creating a god-repository. The `TenantInterceptor` bypasses the repository layer entirely for DB lookups and embeds hardcoded routing logic. UI-specific constants (`DICEBEAR_CDN_DOMAIN`) leak into the API controller.

### Recommendations
| priority | recommendation | effort |
|----------|---------------|--------|
| 1 | Extract org-related methods from `UserRepository` into `OrgRepository` (or a new `MembershipRepository`). `UserService.softDelete()` should delegate org transfer/deletion to `OrganizationService` rather than the user repository. | Medium (2-3d) |
| 2 | Introduce a `TenantRepository` (or `OrganizationLookupRepository`) for `TenantInterceptor.lookupOrganization()`. The interceptor should call a repository, not execute raw Drizzle queries. | Small (1d) |
| 3 | Move `DICEBEAR_CDN_DOMAIN` and `AVATAR_STYLES` validation out of `user.controller.ts`. The API should accept any valid URL; the frontend should enforce DiceBear constraints if desired. | Small (2h) |
| 4 | Move `enforceDeletedOrgRestriction` logic out of `TenantInterceptor`. Consider a guard or middleware that reads allowed routes from a config-driven source, or shift the check to the `OrganizationService` layer. | Small (1d) |
| 5 | Create a cross-domain transaction coordinator (e.g., `AccountDeletionService`) to orchestrate user + org operations. This removes the need for `UserRepository` to know about org tables. | Medium (2-3d) |
| 6 | `UserModule` should explicitly import `TenantModule` if it intends to participate in the tenant context (or document why it is intentionally excluded). | XSmall (1h) |
