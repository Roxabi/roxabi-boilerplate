### Summary
P4 (Consent, GDPR, Audit, Purge) has clean controller->service->repository layering in `consent` and `audit`, but `gdpr` and `purge` services bypass the repository pattern entirely and directly inject `DRIZZLE` to query/update/delete across multiple tables. Additionally, `consent_records` lacks tenant scoping in a multi-tenant SaaS.

### Findings
| severity | file | line | description |
|---|------|------|-------------|
| P1 | apps/api/src/gdpr/gdpr.service.ts | 73 | `GdprService` directly injects `DRIZZLE` and performs inline DB queries across 7 tables (`users`, `sessions`, `accounts`, `members`, `organizations`, `invitations`, `consentRecords`) instead of delegating to repositories. Violates controller->service->repository pattern. |
| P1 | apps/api/src/purge/purge.service.ts | 13 | `PurgeService` directly injects `DRIZZLE` and queries `users`/`organizations` and updates/deletes `members`, `invitations`, `roles` inline instead of using repositories. Only user anonymization is delegated to `UserPurgeService`. |
| P2 | apps/api/src/gdpr/gdpr.controller.ts | 25 | Controller accesses `process.env.APP_NAME` directly instead of using injected `ConfigService`. Inconsistent with `PurgeController` which correctly injects `ConfigService`. |
| P2 | apps/api/src/consent/consent.schema.ts | 10-21 | `consent_records` table has no `organizationId` or tenant-scoped column. In a multi-tenant SaaS, consent is global per-user across all orgs, creating a tenant isolation gap. Inconsistent with `auditLogs` which includes `organizationId`. |
| P3 | apps/api/src/consent/consent.controller.ts | 54-70 | Cookie construction and `reply.setCookie` call are embedded in the controller. HTTP transport concerns should be extracted to a dedicated cookie service or middleware. |

### Recommendations
1. **Extract GDPR repositories** (P1, effort: medium). Create `user.repository.ts`, `session.repository.ts`, `account.repository.ts`, `invitation.repository.ts`, `consent.repository.ts` (read-only) and inject them into `GdprService` instead of `DRIZZLE`. The GDPR export is cross-tenant by design, so repositories can still use `DRIZZLE` with `RLS-BYPASS` if needed.
2. **Extract purge repositories** (P1, effort: medium). Create `userPurge.repository.ts` (already exists via `UserPurgeService` but could be expanded), `organizationPurge.repository.ts` to encapsulate the org update/delete logic. `PurgeService` should orchestrate via repositories, not raw Drizzle.
3. **Use `ConfigService` in `GdprController`** (P2, effort: low). Inject `ConfigService` and replace `process.env.APP_NAME` with `configService.get('APP_NAME')`.
4. **Add tenant scoping to consent** (P2, effort: medium). Evaluate whether `consent_records` should include `organizationId`. If consent is per-user globally, document this in a code comment near the schema definition to clarify the intentional tenant isolation gap.
5. **Extract cookie helper** (P3, effort: low). Move consent cookie payload construction and `setCookie` call to a `ConsentCookieService` or shared cookie utility.
