### Summary
Partition P3 (Admin, system-settings, feature-flags) has a circular dependency between `admin` and `feature-flags`, a god `AdminModule`, several TOCTOU race conditions in member/org services, and DB-level error handling leaking into controllers. No @repo/types boundary violations or direct DB access in controllers were found.

### Findings
| severity | file | line | description |
|----------|------|------|-------------|
| P1 | apps/api/src/feature-flags/repositories/drizzleFeatureFlags.repository.ts | 5 | Circular dependency: `feature-flags` repository imports `FeatureFlagCreateFailedException` from `admin/exceptions/`. `AdminModule` already imports `FeatureFlagsModule`. Repositories should not depend on another module's domain exceptions. |
| P2 | apps/api/src/admin/adminFeatureFlags.controller.ts | 23, 91-99 | Controller imports `PG_UNIQUE_VIOLATION` and catches DB-level unique-constraint errors from `featureFlagService.create`. PostgreSQL error codes should be handled inside the service, not the controller. |
| P2 | apps/api/src/admin/adminMembers.service.ts | 228-247 | `removeMember` reads owner count and then deletes the member outside a transaction. TOCTOU race allows deleting the last owner if a concurrent request removes another owner between the count and the delete. |
| P2 | apps/api/src/admin/adminOrganizations.hierarchy.ts | 46 | `validateHierarchy` calls `getSubtreeDepth(db, orgId)` using the outer `db` connection instead of the transaction `tx`. Breaks read consistency inside the transaction. |
| P2 | apps/api/src/admin/adminOrganizations.service.ts | 148-179 | `createOrganization` checks `getDepth` and then inserts outside a transaction. TOCTOU race allows exceeding max depth if the hierarchy changes between the read and the write. |
| P2 | apps/api/src/admin/admin.module.ts | 23-56 | God module: 7 controllers, 9 providers, 4 module imports. Should be split into submodules (e.g. `admin-users.module.ts`, `admin-orgs.module.ts`) for maintainability and faster unit tests. |

| P3 | apps/api/src/admin/adminFeatureFlags.controller.ts | 140-142 | Controller determines audit action name (`flag.toggled` vs `flag.updated`). Business-rule branching should be in the service. |
| P3 | apps/api/src/admin/filters/adminBadRequest.filter.ts | 5 | Filter imports `SettingValidationException` from `system-settings`. Cross-module filter coupling: admin filters know about system-settings exceptions. |
| P3 | apps/api/src/admin/filters/adminNotFound.filter.ts | 5 | Filter imports `SettingNotFoundException` from `system-settings`. Cross-module filter coupling: admin filters know about system-settings exceptions. |

### Recommendations
1. **Break circular dependency (P1)** — Move `FeatureFlagCreateFailedException` to `feature-flags/exceptions/` or a shared `common/exceptions/` bucket. The repository should throw its own domain exception. Effort: 1h.
2. **Move DB error handling into service (P2)** — Have `FeatureFlagService.create` catch `PG_UNIQUE_VIOLATION` and throw `FlagKeyConflictException`. Remove the try/catch in `AdminFeatureFlagsController`. Effort: 30m.
3. **Wrap race-sensitive operations in transactions (P2)** — Wrap `removeMember` and `createOrganization` in `db.transaction(...)`. Pass `tx` to `getDepth` and `getSubtreeDepth`. Effort: 2h.
4. **Split AdminModule (P2)** — Extract `AdminUsersModule`, `AdminOrganizationsModule`, `AdminMembersModule`, `AdminAuditLogsModule`, and import them into a thin `AdminModule`. Effort: 3h.
5. **Move presentation transforms out of controllers (P3)** — Add `getAllGroupedByCategory()` to `SystemSettingsService` and `determineAuditAction` to `FeatureFlagService`. Effort: 1h.
6. **Decouple filters (P3)** — Move generic exception filters to `common/filters/` or have each module register its own filter via `@UseFilters` on the relevant controller. Effort: 2h.
