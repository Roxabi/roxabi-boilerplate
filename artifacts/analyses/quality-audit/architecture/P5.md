### Summary
P5 partition shows clean NestJS layering: no direct DB access in controllers, no @repo/types boundary violations, no circular dependencies within the partition, and controllers correctly delegate to domain services. Three medium-severity coupling/gaps exist: V1 invitations controller imports a service-internal TTL constant, V1 members controller is structurally coupled to the service return shape, and the V1 exception filter bypasses the global observability filter (missing correlation-id headers and structured logging).

### Findings
| severity | file | line | description |
|----------|------|------|-------------|
| P2 | apps/api/src/v1/controllers/v1Invitations.controller.ts | 16 | Controller imports `INVITATION_TTL_MS` from `../../admin/adminInvitations.service.js` — couples the controller layer to an internal implementation constant of the service layer. The service should expose `invitedAt` directly so the controller does not need the TTL constant. |
| P2 | apps/api/src/v1/controllers/v1Members.controller.ts | 67-75 | Controller maps raw `AdminMembersService.listMembers` return shape fields (`member.user.name`, `member.user.email`) without a typed contract/DTO between service and controller. Inline comment acknowledges the structural coupling risk. |
| P2 | apps/api/src/v1/filters/v1Exception.filter.ts | 26 | `@Catch()` without arguments intercepts all exceptions for V1 routes, fully bypassing the global `AllExceptionsFilter` (apps/api/src/common/filters/allExceptions.filter.ts). The V1 filter omits `x-correlation-id` headers and structured request logging that the global filter provides, creating an observability gap for the public API surface. |
| P3 | apps/api/src/v1/controllers/v1Invitations.controller.ts | 25-28 | `createInvitationSchema` defined inline in the controller instead of in a dedicated `dto/` file, mixing validation definitions with route handling. |
| P3 | apps/api/src/v1/controllers/v1Members.controller.ts | 27-31 | `changeRoleSchema` and `MAX_PAGE_LIMIT` defined inline in the controller instead of in a dedicated DTO/pagination file, mixing validation definitions with route handling. |

### Recommendations (prioritized, with effort estimate)
1. **Add typed return interface for `AdminMembersService.listMembers`** (P2) — 15 min. Define a `ListMemberResult` interface in the service file and have the controller consume it, eliminating the structural coupling noted in the inline comment.
2. **Expose `invitedAt` from `AdminInvitationsService`** (P2) — 20 min. Return `invitedAt` directly from `listPendingInvitations` and `inviteMember` so the controller can drop the `INVITATION_TTL_MS` import and the `mapToInvitationResponse` TTL math.
3. **Enhance `V1ExceptionFilter` with correlation-id and logging** (P2) — 30 min. Inject `ClsService` into the filter, emit `x-correlation-id` headers, and add structured warn/error logs for 4xx/5xx so the V1 surface does not lose observability compared to the rest of the app.
4. **Extract inline Zod schemas to dedicated DTO files** (P3) — 20 min. Move `createInvitationSchema` and `changeRoleSchema` (and `MAX_PAGE_LIMIT`) into `apps/api/src/v1/dto/` to keep controllers thin and consistent with the rest of the codebase.
5. **Remove misleading `@Injectable()` from `ResendEmailProvider`** (P3) — 2 min. Align `ResendEmailProvider` with `NodemailerEmailProvider` by removing the unused decorator since both are instantiated via factory.
