### Summary
Partition P2 (Users, orgs, multi-tenancy) has moderate code-smell density. The most impactful issues are: (1) user repository interface bloat with org-specific methods leaking across module boundaries, and (2) widespread inline `fetch` duplication in admin frontend routes with no shared API client. Backend services are generally well-factored, but several repository methods are duplicated verbatim across the user and organization modules.

### Recommendations (prioritized, with effort estimate)

1. **Extract org-specific repository methods into `OrgRepository` (P1, ~3h)**
   Remove `verifyOrgOwnership`, `verifyTargetMember`, `transferOrgOwnership`, `softDeleteOrg`, `clearOrgSessions`, `expireOrgInvitations` from `UserRepository` and `DrizzleUserRepository`. Inject `OrgRepository` into `UserService` where needed.

2. **Create a shared admin API client in `apps/web/src/lib/admin/api.ts` (P1, ~2h)**
   Wrap all inline `fetch('/api/admin/...')` calls with typed helpers. This eliminates the duplicated error-unwrapping boilerplate across 15 call sites and centralizes credentials/header handling.

3. **Extract `ProfileField` into a shared admin component (P2, ~30m)**
   Move the duplicated component to `components/admin/ProfileField.tsx` and reuse it in both `users.$userId.tsx` and `organizations.$orgId.tsx`.

4. **Extract a shared `TableSkeleton` component (P2, ~30m)**
   Replace `UsersTableSkeleton`, `OrgTableSkeleton`, and `MembersTableSkeleton` with a single configurable `TableSkeleton` in `components/admin/`.

5. **Simplify `UserService.updateProfile` with a field mapper (P2, ~1h)**
   Replace the 8 individual `if` blocks with a helper that maps DTO keys to DB column keys, handling the `fullNameCustomized` logic as a post-mapper step.

6. **Align admin org deletion cleanup with user-facing deletion (P2, ~1h)**
   Add `clearOrgSessions` and `expireOrgInvitations` calls to `AdminOrganizationsDeletionService.deleteOrganization`, or delegate to a shared deletion helper.

7. **Fix `useOrganizations.refetch` invalidation key (P3, ~15m)**
   Change `invalidateQueries` to target the exact key `[...ORGANIZATIONS_QUERY_KEY, sessionKey]` or use `queryClient.invalidateQueries({ queryKey: [...ORGANIZATIONS_QUERY_KEY] })` with `exact: false`.
