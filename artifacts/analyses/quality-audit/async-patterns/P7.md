### Summary
Partition P7 (web components, client libs) has 2 P2 findings where query functions swallow HTTP errors by returning empty arrays instead of throwing, causing React Query to misreport errors as success. The remaining 4 findings are P3: sequential awaits in a loop, floating promises, and empty catch blocks.

### Findings
| severity | file | line | description |
|---|------|------|-------------|
| P2 | apps/web/src/components/admin/PendingInvitations.tsx | 49 | `fetchInvitations` returns `[]` on non-ok instead of throwing. `useQuery` treats the API error as a successful empty response, rendering the empty state instead of the error state. |
| P2 | apps/web/src/lib/useOrganizations.ts | 26 | `queryFn` returns `[]` on non-ok (`res.ok ? res.json() : []`). `useQuery` treats the error as success, masking org-fetch failures. |
| P3 | apps/web/src/lib/shiki.ts | 73-74 | Sequential `await` inside `for...of` loop loads themes one-by-one. Should use `Promise.all(Object.values(options.themes).map(...))` for parallel loading. |
| P3 | apps/web/src/lib/consent/consentProvider.tsx | 139 | `syncToServer(payload)` is called without `await` inside `saveConsent`. The UI completes before the server confirms the sync, which may leave stale consent state if the user navigates away quickly. |
| P3 | apps/web/src/components/settings/PrivacyDataSection.tsx | 32 | Empty `catch` block on GDPR export failure. The loading spinner stops but the user receives no feedback that the export failed. |
| P3 | apps/web/src/lib/profile.ts | 33 | `updateProfile` throws an `Error` with an empty message string when the API error response body lacks a `message` field (`new Error(isErrorWithMessage(data) ? data.message : '')`). This produces unhelpful error messages in the UI. |

### Recommendations (prioritized, with effort estimate)
1. **Fix silent error swallowing in `fetchInvitations` and `useOrganizations`** (P2, 15 min) — Replace `return []` on non-ok with `throw new Error(...)` so React Query surfaces the error state correctly.
2. **Add toast feedback to GDPR export failure** (P3, 5 min) — Replace the empty catch block in `PrivacyDataSection.tsx` with `toast.error(...)`.
3. **Parallelize theme loading in `shiki.ts`** (P3, 10 min) — Replace the sequential `for...of` with `Promise.all(...)`.
4. **Add fallback error message in `updateProfile`** (P3, 5 min) — Use `data.message ?? 'Failed to update profile'` instead of an empty string.
5. **Consider awaiting `syncToServer` or surfacing a sync status** (P3, 30 min) — Either make `saveConsent` async and await `syncToServer`, or add a lightweight sync-status indicator. The current fire-and-forget pattern is acceptable for non-critical data but should be documented as intentional.

### Clean patterns
