# Async Patterns Audit — Partition P1 (AuthN/AuthZ, RBAC, API Keys)

### Summary
One critical unhandled-rejection / race-condition in `auth.instance.ts` where `onOrganizationCreated` is called without `await` inside a Better Auth hook. Two medium-severity issues: a repeated-query anti-pattern inside `seedDefaultRoles` and a split-transaction boundary in the org-creation listener. Two low-severity notes on misleading try/catch and intentional fire-and-forget audit logging.

### Findings

| severity | file | line | description |
|----------|------|------|-------------|
| P0 | `apps/api/src/auth/auth.instance.ts` | 241-246 | `afterCreateOrganization` hook is `async` but calls `onOrganizationCreated(...)` without `await`. The callback returns `Promise<void>` (it runs `eventEmitter.emitAsync`). This creates a floating promise: if the event listener throws, the rejection is unhandled. It also causes a race condition where the org-creation HTTP response is returned before `rbac.listener.ts` has seeded default roles and assigned the Owner role to the creator. |
| P2 | `apps/api/src/rbac/repositories/drizzleRbac.repository.ts` | 184 | `seedDefaultRoles` fetches `allPerms` via `qb.select().from(permissions)` inside the `for (const def of defaultRoles)` loop. With 4 default roles, the same permissions table is queried 4 times identically. The query should be hoisted outside the loop. |
| P2 | `apps/api/src/rbac/rbac.listener.ts` | 23-48 | `handleOrganizationCreated` runs `seedDefaultRoles` (transaction 1) and then a separate `tenantService.queryAs` (transaction 2) for owner assignment. If the server crashes between the two commits, or if the second transaction fails, the organization exists with default roles but the creator is not assigned the Owner role. Missing atomic boundary for org-creation post-processing. |
| P3 | `apps/api/src/auth/auth.guard.ts` | 132-134 | `try/catch` around `this.apiKeyService.touchLastUsedAt(keyData.id)` is misleading. `touchLastUsedAt` returns `void` and handles async errors internally via `.catch()` in `apiKey.service.ts:209`. The `try/catch` will never catch an async error, and no synchronous error can be thrown. |
| P3 | `apps/api/src/api-key/apiKey.service.ts` | 110-124 | `logAudit` uses an intentional fire-and-forget pattern: `this.auditService.log(...).catch(...)` without `await`. Errors are logged but audit records can be silently lost. This is a design choice, but worth explicitly documenting as a known gap. |

### Clean Patterns

| pattern | verdict | notes |
|---------|---------|-------|
| `.map` / `.forEach` with db calls | Clean | All `.map` usage found is in-memory array transformation (perm string resolution, building Maps). No db calls inside `.map` or `.forEach`. |
| `.then` without `.catch` | Clean | No `.then` chains found in target files. |
| `await` inside loops | Clean | Only one loop with `await` found: `seedDefaultRoles` (intentionally sequential). |
| `Promise.all` / parallel db writes | Clean | No `Promise.all` usage in target files. |
| Missing transaction wrapping multi-write | Clean | `rbac.service.ts`, `rbacMember.service.ts` correctly wrap multi-write operations in `tenantService.query`. `apiKey.service.ts` single-write operations do not need transactions. |
| Unawaited `return` in async repositories | Clean | Repository methods that return `qb.select(...)` directly are standard Promise-returning patterns; callers await the result. |

### Recommendations (prioritized, with effort estimate)

1. **Await `onOrganizationCreated` in `auth.instance.ts`** (P0, ~5 min)
   - Change line 242-245 to `await onOrganizationCreated({ organizationId: org.id, creatorUserId: member.userId })`.
   - This ensures Better Auth waits for the full post-creation flow before returning the response, and prevents unhandled rejections.

2. **Hoist `allPerms` query outside the loop in `seedDefaultRoles`** (P2, ~5 min)
   - Move `const allPerms = await qb.select().from(permissions)` to line 167, before the `for (const def of defaultRoles)` loop.

3. **Merge org-creation post-processing into a single transaction** (P2, ~1-2 h)
   - Refactor `seedDefaultRoles` to accept a `tx` parameter and return the created role IDs (or at least the Owner role ID), then perform the owner assignment in the same `tenantService.queryAs` call.
   - Alternatively, add a compensating retry mechanism in the listener for owner assignment failures.

4. **Remove misleading `try/catch` in `auth.guard.ts`** (P3, ~2 min)
   - Remove the `try/catch` around `touchLastUsedAt` (line 132-134). The fire-and-forget contract is already self-documented in the method implementation.

5. **Add explicit `// fire-and-forget` comment to `logAudit`** (P3, ~2 min)
   - Document the intentional async gap in `apiKey.service.ts:110` so future reviewers do not flag it as a bug.
