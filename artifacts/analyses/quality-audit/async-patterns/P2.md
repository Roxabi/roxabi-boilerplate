### Summary
Two P1/P2 issues found in partition P2 (Users, orgs, multi-tenancy): a cache invalidation call placed inside a Drizzle transaction callback risks stale cache state if the transaction rolls back, and a sequential N+1 query loop in the purge repository can degrade performance for users with many deleted organizations. One additional P2 race condition exists in `OrganizationService.reactivate` where a read-check-write sequence lacks a transaction boundary.

### Findings
| severity | file | line | description |
|---|---|---|---|
| P1 | apps/api/src/user/user.service.ts | 173 | `this.invalidateSoftDeleteCache(userId)` is called inside `repo.transaction` callback. If the transaction rolls back after cache invalidation, the in-memory cache becomes stale (user appears not deleted while DB retains the deleted state). Cache invalidation must happen after the transaction commits. |
| P2 | apps/api/src/user/repositories/drizzleUserPurge.repository.ts | 85-102 | Sequential `for...of` loop over `ownedDeletedOrgs` executes 4 queries per org (UPDATE organizations, DELETE members, DELETE invitations, DELETE roles). A user with many deleted orgs triggers N+1*4 queries. Already acknowledged by a TODO comment; still a runtime performance risk. |
| P2 | apps/api/src/organization/organization.service.ts | 55-66 | `reactivate` reads `findOrgForReactivate`, then `checkOwnership`, then writes `reactivateOrg` — none are wrapped in a transaction. A TOCTOU race condition exists: ownership can change between the check and the reactivation, allowing a non-owner to reactivate an organization. |
| P3 | apps/api/src/user/user.service.ts | 90-101 | `updateProfile` reads `getNameFields` then writes `updateProfile` without a transaction. If `fullNameCustomized` flips between the two calls, the auto-derived `name` can overwrite a manually set full name. Low impact (self-inflicted, cosmetic). |

### Recommendations (prioritized, with effort estimate)

1. **Move cache invalidation outside transaction** (`UserService.softDelete`) — ~5 min
   - Relocate `this.invalidateSoftDeleteCache(userId)` to after `await this.repo.transaction(...)` completes, matching the pattern already used in `UserService.purge` (line 213).

2. **Add transaction boundary to `OrganizationService.reactivate`** — ~10 min
   - Wrap `findOrgForReactivate` + `checkOwnership` + `reactivateOrg` in `repo.transaction(async (tx) => { ... })`. Consider adding an ownership check to the `reactivateOrg` UPDATE itself (e.g., `where(and(eq(organizations.id, orgId), eq(members.role, 'owner')))` via a join, or keep the explicit check and reactivation inside the same transaction.

3. **Batch purge operations** (`DrizzleUserPurgeRepository.purgeOwnedOrganizations`) — ~2-4h
   - The TODO already suggests SQL-level random slug generation or a two-pass approach. Short-term: use `Promise.all` for the independent `DELETE` queries per org (members, invitations, roles) to reduce round-trips. Long-term: implement a SQL `UPDATE` with a per-row UUID generated in the database, or batch the org anonymization and the dependent deletes with `inArray()` where possible.

4. **Wrap read-modify-write in transaction for `updateProfile`** — ~10 min
   - Move `getNameFields` inside the same transaction as `updateProfile`, or remove the read path entirely by computing the full name in the frontend before sending the update.
