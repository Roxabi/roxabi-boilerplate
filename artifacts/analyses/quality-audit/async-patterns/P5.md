### Summary
One P1 finding in the queue email handler: a throw inside a batch loop creates a latent duplicate-send risk if `batchSize` is ever increased above 1. All other patterns (await usage, map/forEach, transaction boundaries, promise handling) are clean across the 46 audited source files.

### Findings
| severity | file | line | description |
|----------|------|------|-------------|
| P1 | `apps/api/src/queue/handlers/email.handler.ts` | 18-31 | `handle(jobs)` iterates jobs with `for...of` and `throw error` on first failure. If `batchSize` is ever increased above 1, pg-boss marks the entire batch as failed, so successfully sent emails before the failure are retried and duplicated on retry. Handler should collect per-job outcomes and report partial success/failure instead of throwing mid-batch. |

### Recommendations (prioritized, with effort estimate)

1. **Refactor email handler to per-job completion** (P1, effort: S)
   - Change `EmailQueueHandler.handle` to process each job independently, tracking success/failure per job.
   - Do not throw from the batch handler; instead log failures and return normally so pg-boss marks the batch complete.
   - If per-job retries are needed, use `boss.send` to re-enqueue failed individual jobs.
   - This is the only fix needed to eliminate the latent duplicate-send risk.

2. **Document batch handler contract** (P3, effort: XS)
   - Add a code comment in `email.handler.ts` and `queue.constants.ts` explaining that batch handlers must not throw mid-batch because pg-boss retries the entire batch.
