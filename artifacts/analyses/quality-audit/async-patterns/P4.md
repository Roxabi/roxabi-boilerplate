### Summary
One async-consistency issue found: `consent.controller.ts` applies a side-effect (cookie) before awaiting the DB write, leading to client/DB state mismatch on failure. All other checked patterns (unawaited promises, missing await, N+1 Drizzle in loops, unhandled rejections, blocking sync IO, race conditions, queue/job leaks, missing transaction boundaries) are clean within P4 files.

### Findings

| P2 | apps/api/src/consent/consent.controller.ts | 64 | Side-effect before async: `reply.setCookie()` is executed before `await this.consentService.saveConsent()`. If the DB insert fails (e.g., connection error, constraint violation), the HTTP response still carries the consent cookie while the backend has no record. The frontend will hydrate the banner from the cookie and suppress it, but `GET /api/consent` will 404, causing a silent UI/DB inconsistency. |

Batch size is capped at 100, but this is a bounded N+1-style write pattern. Acceptable for a cron job, but throughput is limited to sequential single-item transactions. |

### Recommendations (prioritized, with effort estimate)
1. **Move `reply.setCookie` after `await saveConsent` in `consent.controller.ts`** (P2, effort: XS, ~5 min). Set the cookie only after the DB record is successfully persisted. For anonymous users (session is null), keep the current flow since there is no DB write. This eliminates the UI/DB mismatch on consent failure.
2. **Consider batching the org purge writes in `purge.service.ts`** (P3, effort: S, ~30 min). The `anonymizedSlug` requires a unique value per org, preventing a simple `inArray` batch update. A two-pass approach could work: first pass bulk-deletes `members`, `invitations`, and `roles` for all expired orgs in a single transaction; second pass loops to assign unique slugs via per-row updates. This halves the transaction count.
3. **Consider batching the user purge writes in `purge.service.ts`** (P3, effort: S, ~30 min). The `userPurgeService.anonymizeUserRecords` does 6 sequential writes per user within a transaction. If the DB supports it, the 5 `delete` operations could be issued in parallel with `Promise.all` within the transaction to reduce latency per user.
