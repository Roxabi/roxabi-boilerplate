# Error-Handling Quality Audit — Partition P3 (Admin, Settings, Feature Flags)

### Summary
Partition P3 has a critical gap: two domain exceptions (`SuperadminProtectionException`, `NotDeletedException`) are not registered in any admin exception filter, causing them to surface as HTTP 500 instead of the correct 4xx status. The web routes contain several swallowed-error patterns and generic `throw new Error` messages that degrade observability and UX. No empty `catch {}` blocks were found in the API layer.

### Recommendations (prioritized, with effort estimate)

1. **Register missing exception filters** (P0) — 30 min
   - Add `SuperadminProtectionException` to `AdminBadRequestFilter` (or create a new `AdminForbiddenFilter` if 403 is desired) and `NotDeletedException` to `AdminBadRequestFilter`. Update all controllers that use these exceptions to include the new filter.

2. **Remove `.catch(() => {})` swallowing in feature-flags.tsx** (P1) — 10 min
   - Remove the `.catch(() => {})` wrappers on lines 140 and 142. The `useMutation` `onError` already surfaces the error via toast; the extra `.catch` suppresses unhandled-rejection telemetry.

3. **Add error logging to `fetchDeletionImpact`** (P1) — 10 min
   - Log the caught error to `console.error` or an error reporter before the `// Proceed without impact summary` fallback.

4. **Standardize web fetch errors** (P2) — 1–2 h
   - Replace all `throw new Error('Failed to fetch ...')` with a helper that extracts the API `message` and `errorCode` (e.g., `throw new ApiError(res.status, body)`). Affected files: `users.tsx`, `audit-logs.tsx`, `system-settings.tsx`, `-organizations-views.tsx`, `organizations.$orgId.tsx`, `users.$userId.tsx`, `-organizations-create-dialog.tsx`.

5. **Parse API error bodies in settings handlers** (P2 → P3) — 30 min
   - In `admin/settings.tsx` `handleDelete`, `handleSave`, and `handleReactivate`, read `res.json()` and show `body.message` when available instead of always falling back to generic i18n keys.

6. **Add `errorComponent` to admin routes** (P3) — 1 h
   - Define a reusable `AdminErrorBoundary` and attach it to all admin routes that currently lack `errorComponent`.

7. **Replace generic `throw new Error` in `drizzleSystemSettings.repository.ts`** (P3) — 15 min
   - Convert the `throw new Error(`Unknown setting type...`)` to a domain exception (e.g., `SettingValidationException`) so it maps to a 400 response instead of 500.
