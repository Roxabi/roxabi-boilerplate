### Summary
Partition P1 (AuthN/AuthZ, RBAC, API keys) has one critical exception-filter gap and several silent catch blocks in the auth hot path. The API key module throws `ApiKeyInsertFailedException` but the registered `ApiKeyExceptionFilter` does not catch it, causing unstructured 500s on DB insert failures. The web auth routes and backend auth callbacks contain numerous bare `catch {}` blocks that log nothing to the console or error reporting service.

### Recommendations
1. **Add `ApiKeyInsertFailedException` to `ApiKeyExceptionFilter`** (effort: 5 min) — append the exception to the `@Catch(...)` list and map it to `HttpStatus.INTERNAL_SERVER_ERROR`.
2. **Add `logger.warn` to `auth.guard.ts` catch block** (effort: 5 min) — log the error object in the `touchLastUsedAt` guard so synchronous failures are observable.
3. **Capture original error in `auth.instance.ts` template catch blocks** (effort: 10 min) — change `catch {}` to `catch (error) {}` and include the error in the `logger.warn` call.
4. **Add `console.error` or Sentry/reporting to web auth catch blocks** (effort: 30 min) — wrap every bare `catch {}` in auth routes with a small helper that shows the toast AND sends the error to the monitoring service.
5. **Add `errorComponent` to auth routes** (effort: 15 min) — provide route-level `errorComponent` on `login.tsx`, `register.tsx`, `dashboard.tsx`, and `verify-email.tsx` for graceful degradation independent of the root boundary.
