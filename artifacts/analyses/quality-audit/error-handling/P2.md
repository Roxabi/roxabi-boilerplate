### Summary
Partition P2 (Users, orgs, multi-tenancy) has one critical silent-failure path in the tenant interceptor that can bypass soft-delete checks, plus five medium and three low findings across API and web.

### Findings
| severity | file | line | description |
|---|---|---|---|
| P1 | apps/api/src/tenant/tenant.interceptor.ts | 118-124 | `resolveParentOrg` catches non-`ForbiddenException` errors, logs them, and returns `orgId` directly. This silently bypasses the soft-delete check and tenant resolution. If `lookupOrganization` fails (e.g., DB timeout or connection error), the interceptor falls back to the raw `activeOrganizationId` as `tenantId`, potentially allowing requests against a deleted organization or with incorrect tenant context. |
| P2 | apps/api/src/user/__tests__/user.integration.spec.ts | 23, 48, 61, 74, 87, 98 | Six `catch (_error)` blocks swallow errors and pass tests unconditionally (`expect(true).toBe(true)`). This makes the integration tests useless for detecting DI wiring failures or module initialization errors. |
| P2 | apps/web/src/components/settings/PrivacyDataSection.tsx | 32-34 | `handleExport` catch block silently swallows GDPR data export failures with no toast, no logging, and no UI state change. The user clicks the button, sees the loading state end, and has no feedback that the download failed. |
| P2 | apps/web/src/routes/admin/settings.tsx | 75-77 | `fetchDeletionImpact` catch block silently swallows errors and returns `null`. The user proceeds to the org deletion confirmation dialog without seeing the impact summary (member count, invitation count, custom role count). |
| P2 | apps/web/src/routes/admin/feature-flags.tsx | 140, 142 | `.catch(() => {})` on `toggleMutation.mutateAsync` and `deleteMutation.mutateAsync` swallows rejections. The mutations already have `onError` toast handlers, but the `.catch` blocks make the promise always resolve, so callers (and TanStack Query's internal state) may treat the operation as successful. |
| P2 | apps/web/src/lib/useOrganizations.ts | 26-27 | `useOrganizations` queryFn returns `[]` on non-ok response without throwing or logging. The user sees an empty organization list instead of an error, which is confusing when the API is actually failing. |
| P3 | apps/web/src/routes/settings/profile.lazy.tsx | 351-353 | `authClient.updateUser` failure is silently swallowed. The inline comment says "avatar will update on next page load", but the user receives a success toast for the profile update and is never informed that the session sync failed. |
| P3 | apps/web/src/lib/profile.ts | 12 | `getProfile` throws a generic `new Error` with only the HTTP status code (`Failed to fetch profile: ${res.status}`). No structured `errorCode` is attached, so consumers cannot distinguish between a 404, 401, or 500. |
| P3 | apps/web/src/routes/admin/members.tsx | 71-72 | `fetchRoles` returns `[]` on non-ok response without throwing or logging. The invite dialog and members table render without role options, silently degrading functionality. |

### Recommendations (prioritized, with effort estimate)

1. **Fix tenant interceptor fallback (P1) — 2h**
   - Change `resolveParentOrg` to throw `DatabaseUnavailableException` (or a new `TenantResolutionException`) instead of returning `orgId` on unexpected errors. Let the registered `DatabaseUnavailableFilter` or global `AllExceptionsFilter` handle the response. Never silently bypass the deleted-org check.

2. **Fix integration test catch blocks (P2) — 1h**
   - Remove the unconditional `expect(true).toBe(true)` catches. Use `.rejects.toThrow()` or `try/catch` with `expect.fail()` so the tests actually validate DI wiring.

3. **Add user feedback to PrivacyDataSection (P2) — 30m**
   - Add `toast.error(m.privacy_export_error())` in the catch block and set an error state so the UI can display a failure message.

4. **Add error feedback to fetchDeletionImpact (P2) — 30m**
   - Either throw the error so the caller can show a toast, or return a discriminated union `{ ok: true, data: DeletionImpact } | { ok: false, error: string }` so the UI can decide whether to show the dialog or an error.

5. **Remove redundant .catch from feature-flags (P2) — 15m**
   - Remove `.catch(() => {})` from the `mutateAsync` calls. The mutations already have `onError` handlers that show toasts. If the child component needs to know the result, pass the promise chain without swallowing, or use `mutate` instead of `mutateAsync`.

6. **Fix useOrganizations silent failure (P2) — 30m**
   - Throw on non-ok response so TanStack Query's `error` state is populated. The component can then render an error UI instead of an empty list.

7. **Add error feedback to profile session sync (P3) — 15m**
   - Show a warning toast in the catch block: "Profile saved, but avatar preview may not update until you refresh."

8. **Add errorCode to profile fetch errors (P3) — 30m**
   - Either read the API error body for `errorCode`, or attach a client-side error code to the thrown Error so the UI can branch on it.

9. **Fix fetchRoles silent failure (P3) — 15m**
   - Throw on non-ok response so the `useQuery` hook surfaces the error, or at least log the failure to the console.
