### Summary
`V1ExceptionFilter` catches all exceptions for v1 controllers but never logs them, causing 500 errors to go unlogged. Two additional P2 issues involve silently swallowed DB migration errors and lost diagnostic context in cursor pagination. No empty catch blocks or missing `@Catch` decorators were found.

### Findings
| severity | file | line | description |
|----------|------|------|-------------|
| P1 | apps/api/src/v1/filters/v1Exception.filter.ts | 28-58 | `V1ExceptionFilter` catches all exceptions via `@Catch()` but does not log them. It formats the response envelope without ever writing to a `Logger`. Because the filter is applied at controller level (`@UseFilters(V1ExceptionFilter)` on all v1 controllers), it intercepts exceptions before the global `AllExceptionsFilter` (which logs) gets a chance. This means 500 errors in the v1 API are silently swallowed from a server-side logging perspective. |
| P2 | apps/api/src/database/database.module.ts | 106 | Bare `catch` block (no error parameter) in `checkPendingMigrations` silently swallows all query failures for `drizzle.__drizzle_migrations`. The comment assumes the table is missing, but if the query fails for any other reason (e.g., permission denied, connection lost, RLS violation), the error is silently swallowed and `applied = 0` is assumed. No `logger.warn` or `logger.error` is emitted. |
| P2 | apps/api/src/common/utils/cursorPagination.util.ts | 49-54 | The `catch` block in `decodeCursor` catches all `Error` instances and re-throws `new InvalidCursorError()` without preserving the original error message or cause. The diagnostic context (e.g., "malformed Base64 or JSON", "missing required fields") is lost, and callers always receive the generic "Invalid cursor token" message. |
| P3 | apps/api/src/common/utils/cursorPagination.util.ts | 66 | `buildCursorCondition` throws a generic `new Error('Failed to build cursor condition')` outside any try/catch. This will bubble up as an unmapped 500 instead of a 400 BadRequest. Should be `new InvalidCursorError('Failed to build cursor condition')` or similar. |
| P3 | apps/api/src/config/env.validation.ts | 184 | `console.error` is used for the security warning about `RATE_LIMIT_ENABLED=false` in production. This bypasses NestJS structured logging, log level configuration, and correlation ID tracking. |
| P3 | apps/api/src/common/url.util.ts | 16 | Bare `catch` in `buildFrontendUrl` silently swallows URL parsing errors without logging. While returning the original URL is intentional, malformed URL inputs are silently ignored with no audit trail. |
| P3 | apps/api/src/queue/handlers/emailDlq.handler.ts | 22-33 | `handle` iterates DLQ jobs and calls `eventEmitter.emit` without a try/catch. If an event listener throws synchronously, the error bubbles up, stops the loop, and causes the DLQ handler to fail. |

### Recommendations
1. **Add logging to V1ExceptionFilter** (P1, ~30 min) — Inject a `Logger` into `V1ExceptionFilter` and log all exceptions, especially 5xx errors with stack traces. Alternatively, remove the `@Catch()` from `V1ExceptionFilter` and make it a response-formatting helper called by `AllExceptionsFilter` for v1 routes, ensuring the global filter still logs.
2. **Log non-missing-table errors in database.module.ts** (P2, ~15 min) — In the bare `catch` at line 106, inspect the error. If it's not "relation does not exist", log it at `error` level before setting `applied = 0`.
3. **Preserve cursor decode context** (P2, ~15 min) — Pass the original error message or cause into the `InvalidCursorError` constructor, e.g., `throw new InvalidCursorError(error.message)` or `throw new InvalidCursorError(undefined, { cause: error })`.
4. **Use InvalidCursorError in buildCursorCondition** (P3, ~10 min) — Replace `throw new Error('Failed to build cursor condition')` with `throw new InvalidCursorError('Failed to build cursor condition')`.
5. **Replace console.error with NestJS Logger** (P3, ~10 min) — In `env.validation.ts`, replace `console.error` with a `Logger` call or pass the warning up to the caller for proper logging.
6. **Add logging to url.util.ts** (P3, ~5 min) — Log malformed URL inputs at `debug` level before returning the original URL.
7. **Wrap DLQ handler event emission in try/catch** (P3, ~15 min) — Wrap each `eventEmitter.emit` call in a try/catch inside `emailDlq.handler.ts` so that one failing listener does not break the entire DLQ batch.
