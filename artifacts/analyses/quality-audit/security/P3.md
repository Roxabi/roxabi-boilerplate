### Summary
P3 (Admin, settings, feature flags) has strong RBAC and RLS-bypass discipline with explicit tenant/org scoping, but contains one high-severity issue: `banExpires` is never checked at authentication time, allowing time-banned users to remain fully authenticated. Also, audit log fire-and-forget errors are silently swallowed and the audit log search over `resourceId` is a text column rather than UUID, causing potential missed results on UUID lookups. No SQL injection, raw sql, eval, secrets, or XSS vectors found.

### Findings
| severity | file | line | description |
|----------|------|------|-------------|
| P1 | apps/api/src/admin/adminUsers.lifecycle.ts | 38-88 | `banUser` sets `banExpires` but the AuthGuard and session enrichment never check `banExpires` against current time. A banned user with an expiry in the future is free to call any API; the ban is only informational. The `banned` flag alone is insufficient for time-based bans. |
| P2 | apps/api/src/admin/adminAuditLogs.service.ts | 79-86 | Audit log `search` filter uses `ilike` on `auditLogs.resourceId` (line 83). `resourceId` is a text column (not UUID). Searching for a UUID substring with `ilike` can match partial UUIDs across unrelated resources, causing false positives or leaking cross-tenant resource IDs in results. |
| P2 | apps/api/src/admin/utils/logAudit.ts | 30-35 | `logUserAudit` and `logOrgAudit` use fire-and-forget `.catch()` that only logs to the application logger. A failed audit log write (e.g., DB unavailable) is never surfaced to the caller or retried. Critical admin actions could proceed with no audit trail. |
| P2 | apps/api/src/admin/adminSettings.controller.ts | 74-99 | `batchUpdate` accepts `z.unknown()` for each setting value. While `SystemSettingsService.validateSettingValue` checks primitive types, any object/array value passes type validation. No structural validation or schema check per setting key exists, enabling arbitrary JSON injection into system settings. |
| P3 | apps/api/src/admin/adminFeatureFlags.controller.ts | 91-99 | `createFlag` catches `PG_UNIQUE_VIOLATION` by checking `(error as Record<string, unknown>).code`. This is a type assertion that will throw a runtime error if `code` is undefined, bypassing the `FlagKeyConflictException` and leaking the raw Postgres error to the client. |
| P3 | apps/api/src/admin/adminOrganizations.controller.ts | 63-73 | `listOrgsQuerySchema.limit` uses `z.preprocess` with `Math.floor(n)` on raw string input. `Number('1e309')` produces `Infinity`, `Math.floor(Infinity)` returns `Infinity`, and `Math.min(Infinity, 100)` returns `100`. But `Number('0x10')` returns `16` (hex), and `Number('')` returns `0` (not NaN). Edge-case parsing quirks. |
| P3 | apps/api/src/admin/adminUsers.controller.ts | 43-54 | `listUsersQuerySchema.limit` has same `z.preprocess` parsing edge cases as org list (see above). `Number('')` is `0`, `Math.floor(0)` is `0`, `Math.min(Math.max(0, 1), 100)` is `1`, so empty string unexpectedly yields limit=1 not the default 20. |

### Recommendations
| priority | item | effort | rationale |
|----------|------|--------|-----------|
| 1 | Add `banExpires` check in AuthGuard | ~2h | Check `banExpires` against `Date.now()` in `runPostAuthChecks` (or `checkSoftDeleted`). If `banned === true` and `banExpires` is in the future (or null = permanent), reject. If `banExpires` is past, auto-unban or clear. This is the only P1 and directly affects authentication correctness. |
| 2 | Add audit log retry + alert | ~3h | Convert fire-and-forget `.catch()` to at-least-once delivery with a bounded retry queue (e.g., BullMQ or in-memory retry with exponential backoff). After max retries, alert ops. Critical for compliance. |
| 3 | Add per-setting JSON schema validation | ~4h | Store a JSON schema per setting key in `systemSettings.metadata`. Validate incoming `value` with `zod` or `ajv` against that schema before writing. Prevents arbitrary object injection. |
| 4 | Fix `resourceId` ilike search | ~1h | Either (a) change `resourceId` to UUID type in DB schema if all resource IDs are UUIDs, or (b) add an exact-match `eq` branch in the audit log search when `search` looks like a UUID (`/^[0-9a-f-]{36}$/i`). |
| 5 | Harden `z.preprocess` limit parsing | ~1h | Replace `Number(val)` with `Number.parseInt(val, 10)` or use `z.coerce.number().int().min(1).max(100)` directly. Prevents hex, Infinity, and empty-string edge cases. |
| 6 | Harden `PG_UNIQUE_VIOLATION` check | ~1h | Replace `(error as Record<string, unknown>).code` with `typeof error === 'object' && error !== null && 'code' in error && typeof error.code === 'string'` guard. Prevents runtime type assertion failure. |
