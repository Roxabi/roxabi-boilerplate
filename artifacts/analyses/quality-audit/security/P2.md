### Summary
Partition P2 (users, orgs, multi-tenancy) has 1 P1 high finding (missing ownership check on deletion-impact endpoint), 1 P1 high finding (members table lacks `deletedAt` causing soft-deleted users to remain in orgs), and 1 P1 high finding (RLS helper function is defined in migrations but never applied to any table, leaving RLS as a no-op). Remaining findings are P2/P3 scoping and defense-in-depth gaps.

### Findings

| severity | file | line | description |
|----------|------|------|-------------|
| P1 | apps/api/src/organization/organization.controller.ts | 61 | `getDeletionImpact` endpoint uses `@Permissions('organizations:delete')` but does NOT verify the requesting user is an owner of the org. `userId` is never passed to `OrganizationService.getDeletionImpact`, so any user with `organizations:delete` permission can query the impact of any org. |
| P1 | apps/api/src/database/schema/auth.schema.ts | 119 | `members` table lacks a `deletedAt` column. When a user is soft-deleted, their `members` records remain active. Queries such as `getOwnedOrganizations`, `verifyOrgOwnership`, and `verifyTargetMember` operate on stale memberships. |
| P1 | apps/api/src/database/schema/auth.schema.ts | 16 | `users` table lacks `tenantId`. The `tenantColumn` from `base.ts` is not used in `auth.schema.ts`. The `organizations`, `members`, `sessions`, `accounts`, `verifications`, and `invitations` tables also lack `tenantId`. The RLS policy `tenant_id = current_setting('app.tenant_id', true)` cannot be applied to these tables. |
| P1 | apps/api/drizzle/migrations/0000_baseline.sql | 217 | `create_tenant_rls_policy(table_name text)` is defined but never called in any migration. No table currently has `ALTER TABLE ... ENABLE ROW LEVEL SECURITY` or `CREATE POLICY tenant_isolation_...`. RLS is a no-op at the DB level. |
| P2 | apps/api/src/user/repositories/drizzleUser.repository.ts | 150 | `verifyOrgOwnership` does not filter by `whereActive(organizations)` or `whereActive(users)`. During `softDelete` user flow, it may return a membership for an already-deleted org. |
| P2 | apps/api/src/user/repositories/drizzleUser.repository.ts | 164 | `verifyTargetMember` does not filter by `whereActive(users)`. During ownership transfer, it may target a soft-deleted user. |
| P2 | apps/api/src/user/repositories/drizzleUser.repository.ts | 129 | `getOwnedOrganizations` does not filter by `whereActive(users)` or `whereActive(members)`. Since `members` has no `deletedAt`, a soft-deleted user still appears as an owner. |
| P2 | apps/api/src/user/user.controller.ts | 29 | `orgResolutionSchema` uses `z.string().min(1)` for `organizationId` and `transferToUserId` instead of `z.string().uuid()`. Non-UUID strings can be passed as IDs. |
| P2 | apps/api/src/user/user.controller.ts | 41 | `deleteAccountSchema` does not limit `orgResolutions` array length. A large array could cause a DoS during the sequential transaction loop in `UserService.softDelete`. |
| P2 | apps/api/src/tenant/tenant.interceptor.ts | 118 | `resolveParentOrg` catches non-ForbiddenException errors, logs them, and falls back to returning the raw `orgId`. If parent resolution fails, the tenant context is set to `orgId` instead of failing closed. |
| P2 | apps/api/src/user/repositories/drizzleUser.repository.ts | 178 | `transferOrgOwnership` does not verify the target user is active. A soft-deleted user could be promoted to owner. |
| P2 | apps/api/src/user/repositories/drizzleUserPurge.repository.ts | 67 | `purgeOwnedOrganizations` deletes `roles` via raw `qb` (bypassing `TenantService.queryAs`). If `roles` ever has RLS force-enabled, the `DELETE` would fail because `app.tenant_id` is not set. |
| P3 | apps/api/src/user/user.controller.ts | 123 | `response.clearCookie('better-auth.session_token', { path: '/' })` hardcodes the cookie name. If the auth library changes the cookie name, the purge endpoint will not clear the session. |
| P3 | apps/api/src/tenant/tenant.interceptor.ts | 37 | `ORG_DELETED_ALLOWED_PATTERNS` includes `GET /api/organizations/:id` but `OrganizationController` has no such endpoint. Dead code. |
| P3 | apps/api/src/tenant/tenant.integration.test.ts | 25 | `process.env.DATABASE_URL` is read directly in a test file. Test-only, not a production leak. |
