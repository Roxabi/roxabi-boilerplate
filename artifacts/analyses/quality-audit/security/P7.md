### Summary
No critical or high security issues found in P7 (Web components, client libs). The codebase demonstrates good security hygiene: no eval, no raw SQL, no hardcoded secrets, React default XSS protection, validated environment variables, and cookie-based auth with `SameSite=Lax` + `Secure` flags.

### Findings
| severity | file | line | description |
|---|------|------|-------------|
| P2 | `apps/web/src/components/admin/UserActions.tsx` | 33-79 | Missing `credentials: 'include'` in authenticated admin fetch calls (`ban`, `unban`, `delete`, `restore`). Same-origin defaults send cookies, but this is inconsistent with the codebase pattern (stated in `api.ts`/`apiKeys.ts`) and with other admin components (`UserContextMenu.tsx`, `MemberContextMenu.tsx`, `OrgListContextMenu.tsx`). Risk: if API origin changes, admin requests would silently fail auth. |
| P2 | `apps/web/src/components/admin/OrgActions.tsx` | 19, 33, 117 | Missing `credentials: 'include'` in authenticated admin fetch calls (`delete org`, `restore org`, `deletion impact`). Same inconsistency as `UserActions.tsx`. |
| P3 | `apps/web/src/lib/routePermissions.ts` | 67 | `createServerFn` handler uses `process.env.API_URL` directly instead of the validated `env.API_URL` from `env.server.ts`. While `envSchema` validates at startup, the direct access bypasses the project's centralized validation pattern. |
