### Summary
P1 (AuthN/AuthZ, RBAC, API keys) shows 1 critical tenant-isolation breach and 2 high severity issues. The most critical finding is `findCandidatesByLastFour` returning API key metadata (including `keyHash` and `keySalt`) across all tenants without tenant scoping. Several RBAC repository methods also rely solely on PostgreSQL RLS without explicit `tenantId` WHERE clauses, creating a defense-in-depth gap if RLS is bypassed or misconfigured.

### Findings
| severity | file | line | description |
|----------|------|------|-------------|
| P0 | apps/api/src/api-key/repositories/drizzleApiKey.repository.ts | 88-106 | `findCandidatesByLastFour` queries `apiKeys` inner-joined with `users` filtered only by `lastFour` and `users.deletedAt IS NULL`. No `tenantId` filter. Returns `keyHash`, `keySalt`, `tenantId`, `scopes`, `role` across all tenants. This is a direct tenant isolation breach and allows cross-tenant API key metadata enumeration. |
| P1 | apps/api/src/rbac/rbacMember.service.ts | 69-106 | `changeMemberRole` allows any member with `members:write` permission to promote any member (including themselves) to the `owner` role. There is no guard restricting the `owner` role assignment to the current owner only. |
| P1 | apps/api/src/rbac/repositories/drizzleRbac.repository.ts | 48, 54, 68, 83, 107, 142 | `findRoleById`, `updateRole`, `deleteRole`, `getRolePermissions`, `listRolesWithPermissions`, and `reassignMembersToRole` lack explicit `tenantId` in their WHERE clauses. They rely entirely on PostgreSQL RLS (via `TenantService.query` transaction). If RLS is misconfigured or the transaction role is not set correctly, these methods become direct cross-tenant read/write vectors. |
| P1 | apps/api/src/api-key/repositories/drizzleApiKey.repository.ts | 83-86, 108-111 | `markRevoked` and `touchLastUsedAt` update by `id` only with no `tenantId` filter. `ApiKeyService.revoke` calls `findForRevoke` first (which does filter by `tenantId`), but `markRevoked` is a separate unscoped call. If invoked directly or from a future code path, it could affect keys across tenants. |
| P2 | apps/api/src/auth/auth.controller.ts | 26-28 | `@All('api/auth/*')` with `@AllowAnonymous()` forwards all Better Auth sub-routes through the NestJS controller, completely bypassing the NestJS guard stack (`@Permissions`, `@Roles`, `@RequireOrg`). Better Auth handles its own session auth, but this creates a security boundary gap where NestJS-side enforcement is absent for the entire `/api/auth/*` surface. |
| P2 | apps/api/src/rbac/permission.service.ts | 17 | `PermissionService` constructor is annotated `// RLS-BYPASS: RBAC resolution` and injects `DRIZZLE` directly, bypassing `TenantService.query()` / `queryAs()`. While current queries use explicit `userId` and `organizationId` filters, the pattern is fragile: any future query added without explicit scoping will leak cross-tenant data. |
| P3 | apps/api/src/api-key/apiKey.service.ts | 182-186 | In `validateBearerToken`, when no candidates match `lastFour`, a constant-time dummy is computed but then compared with itself via `timingSafeEqual(dummy, dummy)`. This is a no-op comparison and does not simulate a real hash verification against a synthetic stored hash. |
| P3 | apps/api/src/auth/auth.instance.ts | 25 | `process.env.APP_NAME` is read directly with a fallback. This bypasses the project's `ConfigService` validation layer, making the env var untracked and untyped. |

### Recommendations
1. **Scope `findCandidatesByLastFour` by tenant** (P0, high effort): Redesign API key token format to embed a tenant prefix or hash, or require a tenant hint in the Authorization header. If the token must remain opaque, at minimum add a secondary index on `(tenantId, lastFour)` and pass the inferred tenant from the request context. As an immediate mitigation, add a `tenantId` filter to the query using the `activeOrganizationId` from the request context when available (for re-validation flows), or shard the lookup.
2. **Restrict Owner role assignment** (P1, low effort): In `RbacMemberService.changeMemberRole`, add a check that rejects promoting a member to `owner` unless the current actor is the existing owner. Alternatively, remove the deprecated endpoint and enforce ownership transfer only via `transferOwnership`.
3. **Add explicit `tenantId` to RBAC repository methods** (P1, medium effort): Update `findRoleById`, `updateRole`, `deleteRole`, `getRolePermissions`, `listRolesWithPermissions`, and `reassignMembersToRole` to include `tenantId` in their WHERE clauses. The repository interface should accept `tenantId` as a required parameter for all role-scoped operations.
4. **Add explicit `tenantId` to API key mutation methods** (P1, low effort): Update `markRevoked` and `touchLastUsedAt` to accept and filter by `tenantId`. Update `ApiKeyService.revoke` to pass `orgId` to `markRevoked`.
5. **Audit `/api/auth/*` surface** (P2, medium effort): Document the explicit guard bypass in `auth.controller.ts`. Add an inline security comment or ADR explaining that Better Auth handles its own auth. Consider registering a lightweight NestJS interceptor on the wildcard route to log/analyze traffic to `/api/auth/*`.
6. **Harden `PermissionService` tenant access pattern** (P2, low effort): Add a code comment or a lint rule enforcing that all queries in `PermissionService` must include both `userId` and `organizationId` filters. Consider a wrapper method that enforces these filters at compile time.
7. **Fix timing-safe dummy comparison** (P3, low effort): Replace `timingSafeEqual(dummy, dummy)` with a comparison against a synthetic stored hash of the same length (e.g., `timingSafeEqual(computedDummy, Buffer.from(randomBytes(64).toString('hex'), 'hex'))`).
8. **Use `ConfigService` for `APP_NAME`** (P3, low effort): Replace `process.env.APP_NAME` with `configService.get<string>('APP_NAME', 'App')` in `auth.instance.ts`.
