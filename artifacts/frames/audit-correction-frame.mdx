## 1. Root Cause Analysis — 5 Systemic Failures

### SF1: Tenant Isolation Collapse
RLS is defined but never applied. `create_tenant_rls_policy` exists in the baseline migration yet no table has `ALTER TABLE ... ENABLE ROW LEVEL SECURITY`. The defense-in-depth strategy assumed RLS would be the sole tenant boundary, so repository methods omitted explicit `tenantId` WHERE clauses. When RLS is a no-op, cross-tenant read/write vectors open. The `TenantInterceptor` compounds this by silently falling back to raw `orgId` on DB errors instead of failing closed.

### SF2: Error Swallowing & Unstructured Error Handling
The frontend adopted a pattern of `catch {}` with `toast.error(m.auth_toast_error())` — the user sees a generic message while the original error (and its stack/cause/status code) is lost. Backend exception filters are incomplete: `SuperadminProtectionException` and `NotDeletedException` are thrown but never registered in `@Catch`, causing them to surface as HTTP 500. `V1ExceptionFilter` intercepts all exceptions for v1 routes but never logs them, creating an observability blind spot.

### SF3: Architecture Boundary Violations
`UserRepository` is a god-repository touching 5 domain schemas (`users`, `organizations`, `members`, `invitations`, `sessions`, `roles`). `GdprService` and `PurgeService` bypass the repository layer entirely, injecting `DRIZZLE` and querying 7 tables inline. `AdminModule` accumulates 7 controllers and 9 providers. A circular dependency exists between `feature-flags` and `admin` modules. UI-specific constants (`DICEBEAR_CDN_DOMAIN`, `AVATAR_STYLES`) leak into API controllers. The frontend `lib/consent/consentProvider.tsx` imports from `components/`, creating a circular dependency across the lib/components boundary.

### SF4: Async Discipline & Transaction Boundary Failures
`onOrganizationCreated` is called without `await` inside a Better Auth `afterCreateOrganization` hook — a floating promise that causes race conditions and unhandled rejections. Cache invalidation is placed inside a Drizzle transaction callback; if the transaction rolls back, the cache becomes stale. Multiple read-check-write sequences (`removeMember`, `createOrganization`, `reactivate`, `batchUpdate`) operate outside transactions, creating TOCTOU races. `queryClient.invalidateQueries` calls are fire-and-forget in callbacks. The email queue handler throws mid-batch, risking duplicate sends on retry.

### SF5: Config & Validation Failures
Hardcoded `http://localhost:3000` and `http://localhost:4000` fallbacks exist in `auth.service.ts`, `authClient.ts`, `routePermissions.ts`, and `env.validation.ts`. These defaults mask missing env vars at boot time and break in non-localhost deployments. `batchUpdate` accepts `z.unknown()` for system setting values with no per-key schema validation. `orgResolutionSchema` accepts `z.string().min(1)` instead of `z.string().uuid()`. `console.error` calls leak error details in production UI. `window.location.reload()` and `window.location.href` mutations break SPA navigation.

## 3. Risk Assessment

| Phase | What Could Break | What Needs Migration | Rollback Strategy |
|-------|------------------|----------------------|-------------------|
| Phase 1 | RLS migration fails on existing data; `members` schema change breaks existing queries; `TenantInterceptor` fail-closed breaks legitimate parent-org lookups | DB migration + data backfill for `members.deletedAt`; verify all `members` queries include `whereActive` | DB snapshot before migration; migration is reversible via `ALTER TABLE` |
| Phase 2 | Structured errors change frontend error parsing; new `ApiError` class may break existing `catch (e: any)` blocks | Update any frontend code that inspects `error.message` directly; verify all toasts still render correctly | Error classes are additive; old `Error` instances still work; revert individual catch changes if needed |
| Phase 3 | God-repo split breaks DI wiring; new repository interfaces may not cover all existing call sites; module split may break `AdminModule` imports | Update all imports of `UserRepository` org methods; add `MembershipRepository` to `UserModule` providers; verify `AdminModule` sub-modules export same public interface | Add re-exports to `AdminModule` as compatibility layer; keep old `UserRepository` interface for one release with `@deprecated` |
| Phase 4 | Transaction wrapping changes dead-lock behavior; CTE replacement may have different null/empty-set semantics; `await` insertion may delay response timing | Monitor DB lock wait times; verify CTE returns same shape as recursive loop; add timeout to `onOrganizationCreated` | Transaction wrapping is revertible per-method; CTE can be reverted to loop if performance is worse |
| Phase 5 | `getOrThrow` causes boot failure if env vars are missing in existing deployments; `navigate()` changes may break deep-link behavior | Add missing env vars to `.env.example` and deployment config; verify `navigate()` preserves query params | `getOrThrow` → `get` fallback is a one-line revert; `window.location` revert is trivial |

## 4. Decision Points (human approval required)

**Before Phase 1:**
- Approve RLS migration strategy: enable on all tables vs. selective enablement. Does `DATABASE_APP_URL` role exist in production? (If not, RLS enablement will break the app until the role is created.)
- Approve `members.deletedAt` migration: soft-delete existing `members` records or leave `NULL`?
- Confirm `banExpires` semantics: auto-unban when expired, or require manual unban?

**Before Phase 2:**
- Approve frontend error structure: do we want a typed `ApiError` class or a plain `{ status, message, errorCode }` object?
- Confirm scope of `errorComponent` for admin routes: reuse existing `ErrorBoundary` or create a dedicated `AdminErrorBoundary`?

**Before Phase 3:**
- Approve `MembershipRepository` interface: which methods move from `UserRepository`? (`verifyOrgOwnership`, `transferOrgOwnership`, `softDeleteOrg`, `clearOrgSessions`, `expireOrgInvitations` — all of them?)
- Approve `AdminModule` split granularity: 4 sub-modules (users, orgs, members, audit) or more granular?
- Confirm `DICEBEAR_CDN_DOMAIN` removal from API: does the frontend need to enforce avatar URL constraints instead?

**Before Phase 4:**
- Approve CTE replacement strategy for org hierarchy: do we maintain the `MAX_PARENT_WALK_ITERATIONS` safety cap inside the CTE or drop it?
- Confirm `email.handler.ts` batch contract: do we need per-job retry (re-enqueue) or is batch-level retry sufficient?
- Approve `featureFlags` cache stampede protection: add a `Promise` cache in `FeatureFlagService` or use a shared cache layer?

**Before Phase 5:**
- Approve missing-env boot failure: is `getOrThrow` acceptable for all required env vars, or should some have staging-safe defaults?
- Confirm `legal.config.ts` externalization: move to `.env` or `SystemSettings`?
- Approve `window.location` → `navigate()` migration: should `magic-link/verify.tsx` still use full reload for external link verification?

## 7. Phase Dependency Graph

```
Phase 1 (Tenant Isolation)
  ↓
Phase 2 (Error Handling)
  ↓
Phase 3 (Architecture)
  ↓
Phase 4 (Async & Transactions)
  ↓
Phase 5 (Config & Validation)

Quick Wins ∥ any phase
```

**Rationale for sequential ordering:**
- Phase 1 must come first because all later phases touch the same repository and service files; tenant isolation changes are the most critical and must be validated before anything else is layered on top.
- Phase 2 follows because error handling is pervasive; fixing it first prevents the architecture changes in Phase 3 from being debugged against swallowed errors.
- Phase 3 follows because splitting repositories and modules introduces new interfaces and DI wiring; doing this after error handling is stable means any test failures are due to the architecture change, not pre-existing error-surface gaps.
- Phase 4 follows because transaction boundaries depend on the repository structure established in Phase 3.
- Phase 5 comes last because it is the lowest risk and the most mechanical; by this point the codebase is structurally sound and only needs config tightening.
